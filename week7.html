Session 1 :- Session ID Based Login & Caching

Session ID-based login and caching are concepts often employed in web development to enhance security and improve performance.

Session ID-Based Login:

When a user logs into a web application, the server creates a unique identifier known as a Session ID for that user. This ID is usually a random string or a combination of various parameters. The Session ID is then stored on the server side, associating it with the user's session data, which may include information like user credentials, preferences, and other relevant details. The Session ID is also typically sent to the user's browser as a cookie. This allows the user's browser to include the Session ID in subsequent requests to the server, enabling the server to identify and authenticate the user without requiring them to re-enter their credentials for every interaction. The server validates the Session ID with its stored session data to ensure the user is authenticated. If the Session ID is valid, the server allows access to the user's account.

Caching:

Caching involves storing copies of frequently accessed data in a location that allows for faster retrieval. In web development, caching is often implemented to improve the performance and responsiveness of a website or web application by reducing the need to repeatedly fetch data from the server. Common caching strategies include:

1. Page Caching: Storing entire HTML pages so that subsequent requests for the same page can be served quickly without regenerating the content.

2. Object Caching: Storing individual pieces of data, such as database query results or API responses, to avoid redundant computations or database queries.

3. CDN (Content Delivery Network) Caching: Distributing cached content across multiple servers geographically closer to users, reducing latency and improving load times.

Integration of Session ID-Based Login and Caching:

Caching can be combined with session ID-based login to optimize the performance of authenticated sessions. After a user logs in, certain data relevant to their session, such as user-specific content or preferences, can be cached to reduce the need for repeated database queries. However, it's crucial to manage cache expiration and invalidation carefully to ensure that users receive up-to-date information and that sensitive data is not compromised.

Additionally, secure practices must be followed to protect Session IDs from theft or misuse, as they are crucial for maintaining user authentication.

In summary, Session ID-based login enhances user authentication, while caching optimizes performance by reducing the time and resources needed to serve frequently accessed data. Combining these techniques can result in a more secure and responsive web application.

                    Encoding.UTF8.GetBytes(secretKey));

                SigningCredentials credentials = new SigningCredentials(symmetricSecurityKey,
                    SecurityAlgorithms.HmacSha256);

                // Creating claims for the user
                List<Claim> claims = new List<Claim>

                    new Claim("Id", objUser.R01F01.ToString())

                // Creating JWT token with claims and signing credentials
                JwtSecurityToken token = new JwtSecurityToken(issuer, issuer, claims,
                    expires: DateTime.Now.AddMinutes(20),
                    signingCredentials: credentials);

                                        absoluteExpiration: DateTime.MaxValue,
                                        slidingExpiration: slidingExpiration,
                                        CacheItemPriority.Normal,
                                        onRemoveCallback: null);

                BLHelper.ServerCache.Add(sessionId.ToString(),
                                         value: jwtToken,
                                         dependencies: null,
                                         absoluteExpiration: DateTime.MaxValue,
                                         slidingExpiration: slidingExpiration,
                                         priority: CacheItemPriority.Normal,
                                         onRemoveCallback: null);

                BLHelper.ServerCache.Add(userDataSessionId,
                                         value: "Testing",
                                         dependencies: null,
                                         absoluteExpiration: DateTime.MaxValue,
                                         slidingExpiration: slidingExpiration,
                                         priority: CacheItemPriority.Normal,
                                         onRemoveCallback: null);

                //CookieHeaderValue cookieActiveSessionId =
                //    new CookieHeaderValue("Session-Id", sessionId.ToString())
                //    {
                //        Path = "/",
                //        HttpOnly = true,
                //    };

                // Cookies send to user for authentication, active sessioin purpose.
                CookieHeaderValue[] cookies =
                {
                    new CookieHeaderValue("LoginSessionId", sessionId.ToString())
                    {
                        Path = "/",
                        HttpOnly = true
                    },

                        Path = "/",

                    "An error occured during generating token.");
            }
        }

BearerAuthAttribute.cs


    using OnlineShoppingAPI.Business_Logic;
    using System;
    using System.Linq;
    using System.Net;
    using System.Net.Http;
    using System.Net.Http.Headers;
    using System.Security.Principal;
    using System.Web;
    using System.Web.Http.Controllers;
    using System.Web.Http.Filters;
    
    namespace OnlineShoppingAPI.Security
    {
        public class BearerAuthAttribute : AuthorizationFilterAttribute
        {
            public override void OnAuthorization(HttpActionContext actionContext)
            {
                try
                {
                    // Getting Cookie Value
                    CookieHeaderValue cookieJwtToken = actionContext.Request.Headers
                        .GetCookies().FirstOrDefault();
    
                    if (cookieJwtToken["Token"] == null || cookieJwtToken["Token"].Value == "")
                    {
                        actionContext.Response = BLHelper.ResponseMessage(
                            HttpStatusCode.Unauthorized, "Please provide token");
                        return;
                    }
    
                    string token = cookieJwtToken["Token"].Value;
    
                    if (!BLToken.IsJwtValid(token))
                    {
                        actionContext.Response = BLHelper.ResponseMessage(
                            HttpStatusCode.Unauthorized, "Alteration with the Token");
                        return;
                    }
    
                    CookieHeaderValue cookieLoginSessionId = actionContext.Request.Headers
                        .GetCookies("LoginSessionId").FirstOrDefault();
                    string sessionId = cookieLoginSessionId["LoginSessionId"]?.Value;
    
                    string jwtToken = (string)BLHelper.ServerCache.Get(sessionId);
    
                    if (jwtToken == null)
                    {
                        actionContext.Response = BLHelper.ResponseMessage(
                            HttpStatusCode.Unauthorized, "Login expired, please login.");
                        return;
                    }
    
                    if (jwtToken != token)
                    {
                        actionContext.Response = BLHelper.ResponseMessage(
                            HttpStatusCode.Unauthorized, "Token Modificication Error");
                        return;
                    }
    
                    IPrincipal principal = BLToken.GetPrincipal(jwtToken);
    
                    if (principal == null)
                    {
                        actionContext.Response = BLHelper.ResponseMessage(
                            HttpStatusCode.Unauthorized, "Unauthorized user.");
                        return;
                    }
    
                    HttpContext.Current.User = principal;
                }
                catch (Exception ex)
                {
                    BLHelper.LogError(ex);
                    actionContext.Response = BLHelper.ResponseMessage(HttpStatusCode.BadRequest,
                        "An error occured during authentication");
                }
            }
        }
    }

Section 2 :- Git & Github

Git:

Git is a distributed version control system (VCS) designed to handle everything from small to very large projects with speed and efficiency. It was created by Linus Torvalds in 2005. Here are some key concepts and features of Git:

1. Version Control: Git tracks changes in source code during software development. It allows multiple developers to work on a project simultaneously without interfering with each other's work.

2. Distributed System: Git is a distributed VCS, meaning every developer has a complete copy of the repository, including its history. This allows for offline work and provides redundancy.

3. Commits: Developers make changes to the codebase and create a snapshot of these changes, called a commit. Commits are accompanied by a commit message that describes the changes made.

4. Branching: Git allows the creation of branches, which are independent lines of development. Developers can work on a feature or bug fix in a branch without affecting the main codebase until they are ready to merge their changes.

5. Merging: Merging brings changes from one branch (e.g., a feature branch) into another (e.g., the main branch). Git handles merging automatically in many cases but may require manual resolution of conflicts.

6. Repositories: A Git repository is a collection of files and their history. Repositories can be hosted on a local machine or a remote server.

GitHub:

GitHub is a web-based platform that provides hosting for software development and version control using Git. It adds a graphical interface and various collaboration features on top of Git. Here are key aspects of GitHub:

1. Remote Hosting: GitHub serves as a remote repository hosting service for Git repositories. Developers can push their local Git repositories to GitHub to share code and collaborate with others.

2. Collaboration: GitHub facilitates collaboration through features like pull requests, issues, and discussions. Pull requests allow developers to propose changes and request them to be merged into a target branch.

3. Issues and Projects: GitHub provides tools for issue tracking and project management. Issues can be used to track bugs, enhancements, and tasks, while Projects help organize and manage work.

4. Forks and Clones: Users can fork repositories to create their own copy of a project, which can be developed independently. Cloning a repository allows a developer to create a local copy on their machine to work on.

5. GitHub Pages: GitHub Pages is a feature that allows users to publish websites directly from their GitHub repositories.

6. Community and Social Coding: GitHub fosters a strong sense of community among developers. Social coding features, like the ability to follow users, star repositories, and contribute to open source projects, enhance collaboration.

In summary, Git is a version control system, and GitHub is a platform built on Git that provides additional features for collaboration, hosting, and community engagement in software development. Many developers use Git for version control and GitHub for collaboration and hosting their Git repositories.

Section 4 :- Bogus Library (Fake Data Generator)

One such library is the "Bogus" library for C#, commonly used for generating fake data for testing and seeding databases. It allows developers to create realistic-looking but not real data for scenarios where using actual data is unnecessary or impractical.

Here is a simple example of using the Bogus library in C# for generating fake user data:

Bogus Demo


    using Bogus;

    class Program
    {
        static void Main()
        {
            var faker = new Faker<User>()
                .RuleFor(u => u.Id, f => f.IndexFaker)
                .RuleFor(u => u.FirstName, f => f.Person.FirstName)
                .RuleFor(u => u.LastName, f => f.Person.LastName)
                .RuleFor(u => u.Email, (f, u) => f.Internet.Email(u.FirstName, u.LastName));
    
            var fakeUsers = faker.Generate(5);
    
            foreach (var user in fakeUsers)
            {
                Console.WriteLine($"ID: {user.Id}, Name: {user.FirstName} {user.LastName}, Email: {user.Email}");
            }
        }
    }
    
    public class User
    {
        public int Id { get; set; }
        public string FirstName { get; set; }
        public string LastName { get; set; }
        public string Email { get; set; }
    }

In this example, the Bogus library is used to create a Faker instance for generating fake User objects with random names and email addresses. The resulting fake user data can be useful for testing scenarios without using real user information.
